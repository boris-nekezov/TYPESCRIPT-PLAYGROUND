## Stack

React 19, TypeScript, Vite, React Query v5, Zustand, Tailwind CSS

## Architecture

- Feature-based folder structure
- Each component in its own folder: `ComponentName/ComponentName.tsx`, `ComponentName.types.ts`, `ComponentName.test.tsx`
- Shared/reusable components in `src/components/`
- Feature modules in `src/features/`

## Code Style

- Named function components (const arrow), default export at bottom
- Props type imported with `import type` from separate `.types.ts` file
- No React.FC — destructure props directly in function signature
- Tailwind for styling — no CSS modules, no styled-components
- Strict TypeScript — no `any`, no `as` casting unless justified with comment

## Conventions

- Follow Vercel React best practices for component design and data fetching patterns
- Apply composition patterns — prefer render props and compound components over deep prop drilling

## Documentation

- Never rely on training knowledge for library APIs — always fetch fresh docs
- DocsExplorer uses Context7 MCP as primary source with web fallback

## 🚀 Сесия 1 — `type` vs `interface`

### Теория (3 минути)

- **`interface`** — за описване на обекти и класове. Поддържа **declaration merging**.
- **`type`** — по-гъвкав, поддържа **unions**, **intersections**, **primitives**. Не се merge-ва.

```ts
// interface — може да се extend-ва и merge-ва
interface User {
  id: number;
  name: string;
}
interface User {
  email: string; // ✅ merge-ва се автоматично
}

// type — по-гъвкав
type ID = string | number; // unions — само с type!
type Admin = User & { role: 'admin' }; // intersection
```

> **Практическо правило:** `interface` за публичен API/props, `type` за всичко останало.
